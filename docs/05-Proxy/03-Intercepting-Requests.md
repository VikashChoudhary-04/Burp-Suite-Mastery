# Intercepting Requests

## Overview

- Request interception is one of the core capabilities of Burp Proxy.

- It allows you to pause an HTTP or HTTPS request before it reaches the target server.

- While a request is intercepted, you can:

  * Inspect its structure.
  * Review parameters.
  * Examine headers and cookies.
  * Understand authentication context.
  * Identify client-controlled values.
  * Decide whether the request should be forwarded, dropped, modified, or sent to another Burp tool.

- Request interception is most useful when applied deliberately.

- Professional testing does not require intercepting every request.

- A more efficient workflow is usually:

  ```text
  Browse Normally

  ↓

  Review HTTP History

  ↓

  Identify an Interesting Request

  ↓

  Enable Intercept When Needed

  ↓

  Inspect the Request

  ↓

  Decide the Next Testing Action
  ```

## What Is Request Interception?

- Request interception pauses a client request inside Burp Proxy before Burp forwards it to the destination server.

- The normal communication flow is:

  ```text
  Browser
      │
      ▼
  Burp Proxy
      │
      ▼
  Target Server
  ```

- With interception enabled:

  ```text
  Browser
      │
      ▼
  Burp Proxy
      │
      ▼
  Request Paused
      │
      ├── Inspect
      ├── Edit
      ├── Forward
      ├── Drop
      └── Send to Another Tool
  ```

- Until you take an action, the intercepted request remains paused.

---

## Intercept ON vs Intercept OFF

- When **Intercept is ON**:

  * Requests matching the configured interception rules are paused.
  * You can inspect each request before it reaches the server.
  * Browser activity may appear to stop while the request waits in Burp.
  * Multiple requests may queue behind the currently intercepted request.

- When **Intercept is OFF**:

  * Requests pass through Burp automatically.
  * Burp can still record traffic in HTTP history.
  * Normal browsing is faster and less disruptive.
  * Interesting requests can still be selected later for deeper testing.

- A practical professional default is:

  ```text
  Normal Browsing
  → Intercept OFF

  Targeted Investigation
  → Intercept ON When Required
  ```

---

## Why Intercept Requests?

- Browsers normally send requests automatically.

- Interception gives you a controlled observation point before the server processes a request.

- This can help you understand:

  * Which endpoint receives the request.
  * Which HTTP method is used.
  * Which parameters are submitted.
  * Which headers influence behavior.
  * Which cookies maintain session state.
  * Which identifiers reference application objects.
  * Which values appear security-sensitive.
  * Which fields are controlled by the client.

- Interception is especially useful when a request:

  * Is generated only once.
  * Contains short-lived values.
  * Performs a sensitive state-changing action.
  * Is difficult to locate later.
  * Must be inspected before submission.
  * Depends on a particular workflow state.

## Anatomy of an Intercepted Request

- Consider:

  ```http
  POST /account/email HTTP/1.1
  Host: example.com
  Cookie: session=abc123
  Content-Type: application/x-www-form-urlencoded
  Content-Length: 48

  email=alice%40example.com&csrf=9f31ab
  ```

- Important areas include:

  ```text
  Method
      ↓
  POST

  Path
      ↓
  /account/email

  Host
      ↓
  example.com

  Session Context
      ↓
  Cookie: session=abc123

  Request Body
      ↓
  email=...

  Anti-CSRF Value
      ↓
  csrf=9f31ab
  ```

- Before modifying anything, understand what each component appears to represent.

## Establish the Baseline First

- A baseline is the legitimate request generated by normal application behavior.

- Preserve it before testing.

- A strong baseline records:

  * HTTP method.
  * Endpoint.
  * Parameters.
  * Headers.
  * Cookies.
  * Authentication state.
  * User role.
  * Relevant object identifiers.
  * Expected response.
  * Expected final application state.

- The basic principle is:

  ```text
  Capture Legitimate Request

  ↓

  Understand Expected Behavior

  ↓

  Preserve Baseline

  ↓

  Modify One Relevant Variable

  ↓

  Compare Results
  ```

- Editing several values before understanding the baseline makes later conclusions weaker.

---

## Identify Client-Controlled Input

- Intercepted requests reveal values the client sends to the server.

- Common examples include:

  * Query parameters.
  * Form fields.
  * JSON properties.
  * XML values.
  * Cookies.
  * Custom headers.
  * Object identifiers.
  * User identifiers.
  * Tenant identifiers.
  * Role-related values.
  * Prices and quantities.
  * Workflow-state values.
  * File metadata.

- Client-controlled does not automatically mean vulnerable.

- It means the value may deserve investigation if the server relies on it for a security-sensitive decision.

---

## Authentication and Session Context

- Always identify the request's security context.

- Look for:

  * Session cookies.
  * Authorization headers.
  * Bearer tokens.
  * CSRF tokens.
  * API keys.
  * Custom authentication headers.
  * User or tenant identifiers.

- Ask:

  ```text
  Who is making this request?

  What role does the user have?

  Which account or tenant is active?

  What proves authentication?

  What values identify the target object?

  Which controls should the server enforce?
  ```

- Without this context, authorization and session-testing results can easily be misinterpreted.

---

## State-Changing Requests

- Give particular attention to requests that change application state.

- Examples include:

  ```text
  POST /account/email

  PUT /api/profile/123

  PATCH /api/users/42

  DELETE /api/orders/9001

  POST /checkout

  POST /admin/users/create
  ```

- Before interacting with such requests, understand:

  * What action will occur.
  * Which account is affected.
  * Whether the action is reversible.
  * Whether the target is authorized for testing.
  * Whether repeated requests could create unwanted effects.

- Use controlled test data whenever possible.

---

## Intercepting a Request in Burp

- A basic workflow is:

  1. Confirm that the browser is correctly proxied through Burp.
  2. Open **Proxy**.
  3. Open **Intercept**.
  4. Turn interception ON.
  5. Perform the intended action in the browser.
  6. Inspect the paused request.
  7. Decide whether to forward, drop, edit, or send it to another Burp tool.
  8. Turn interception OFF when targeted interception is no longer needed.

- Do not leave interception enabled unnecessarily during general browsing.

---

## Queued Requests

- Modern applications often generate many requests at once.

- When interception is enabled, Burp may capture:

  * Page requests.
  * API requests.
  * JavaScript files.
  * Images.
  * Analytics requests.
  * Background polling.
  * Authentication refresh requests.
  * Third-party resources.

- This can create a queue.

- If the browser appears frozen, check whether Burp is waiting for you to process intercepted requests.

- Avoid blindly forwarding requests without understanding whether they belong to the target workflow.

---

## Background Traffic

- Not every intercepted request is relevant.

- Modern browsers and applications generate background traffic such as:

  * Telemetry.
  * Analytics.
  * Content delivery requests.
  * Browser services.
  * Third-party integrations.
  * Periodic API polling.

- Scope and interception rules help reduce noise.

- A cleaner workflow is:

  ```text
  Define Scope

  ↓

  Browse Normally

  ↓

  Filter Noise

  ↓

  Intercept Relevant Traffic

  ↓

  Investigate Deliberately
  ```

---

## Scope-Aware Interception

- Interception should respect the authorized assessment scope.

- Before testing, confirm:

  * Allowed domains.
  * Allowed subdomains.
  * Allowed application components.
  * Testing restrictions.
  * Prohibited actions.
  * Third-party dependencies that are out of scope.

- A request appearing in Burp does not automatically mean you are authorized to test its destination.

- Configure scope and interception rules to reduce accidental interaction with unrelated systems.

---

## Choosing the Next Action

- After intercepting a request, decide what you actually need to do.

- Common decisions include:

  ```text
  Need the Request to Continue Normally?
      ↓
  Forward

  Request Should Not Reach the Server?
      ↓
  Drop

  Need a Small One-Time Change?
      ↓
  Edit Before Forwarding

  Need Repeated Controlled Testing?
      ↓
  Send to Repeater

  Need Systematic Input Variation?
      ↓
  Send to Intruder

  Need to Compare Data?
      ↓
  Use Comparer

  Need to Decode or Transform a Value?
      ↓
  Use Decoder
  ```

- Interception is therefore often the beginning of an investigation rather than the entire testing workflow.

---

## When Not to Use Interception

- Intercepting every request can reduce efficiency.

- Avoid unnecessary interception when:

  * You are simply browsing the application.
  * HTTP history already contains the request you need.
  * You are mapping application functionality.
  * Background traffic creates excessive noise.
  * You need repeated testing better suited to Repeater.
  * You are performing structured automation using another Burp tool.

- Prefer the simplest tool that accomplishes the testing objective.

## Common Mistakes

- Common interception mistakes include:

  * Leaving Intercept ON during normal browsing.
  * Assuming the browser is broken when requests are queued.
  * Modifying several parameters simultaneously.
  * Testing before preserving a legitimate baseline.
  * Ignoring authentication and session context.
  * Forwarding requests without understanding their effect.
  * Testing third-party or out-of-scope hosts.
  * Treating every unusual response as a vulnerability.
  * Repeating state-changing actions unnecessarily.
  * Using interception for tasks better suited to Repeater.

## Practical Example

- Suppose an authenticated user changes a profile display name.

- The browser sends:

  ```http
  POST /api/profile HTTP/1.1
  Host: example.com
  Cookie: session=abc123
  Content-Type: application/json

  {
    "displayName": "Alice"
  }
  ```

- A disciplined investigation begins by identifying:

  ```text
  Action
      ↓
  Update profile

  Authentication
      ↓
  session=abc123

  Client-Controlled Input
      ↓
  displayName

  Expected State Change
      ↓
  Current user's display name changes
  ```

- Preserve the legitimate request first.

- If deeper testing is required, send the request to Repeater rather than repeatedly intercepting the browser workflow.

## Practice Tasks

1. Configure a browser to use Burp Proxy.

2. Open an authorized lab or deliberately vulnerable application.

3. Keep Intercept OFF and browse normally.

4. Review HTTP history and identify a simple request.

5. Turn Intercept ON.

6. Trigger the request again.

7. Identify:

   * Method.
   * Host.
   * Path.
   * Parameters.
   * Cookies.
   * Authentication information.
   * Client-controlled values.

8. Preserve the original request as a baseline.

9. Forward the request without modification.

10. Observe the normal application result.

11. Repeat the workflow and inspect how queued requests behave.

12. Turn Intercept OFF and confirm normal browsing resumes.

## Key Takeaways

- Request interception gives you control over traffic before it reaches the server.

- Its value comes from deliberate use, not from keeping every request paused.

- Strong testing begins by understanding the legitimate request, authentication context, client-controlled values, and expected application behavior.

- Preserve a clean baseline before experimentation.

- Use interception to capture and understand important requests, then move them to the most appropriate Burp tool for deeper investigation.

- Always operate within explicit authorization and scope.
