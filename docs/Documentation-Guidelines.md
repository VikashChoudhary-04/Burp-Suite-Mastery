# Documentation Guidelines

- This document defines the writing standards used throughout the **Burp Suite Mastery** repository.

- Following these guidelines ensures every module is consistent, easy to read, and practical.

---

## Writing Principles

- Every document should:

  * Explain concepts before procedures.
  * Focus on understanding, not memorization.
  * Be practical and hands-on.
  * Use simple, clear language.
  * Build from beginner to advanced.
  * Reflect professional penetration testing workflows.

---

## Standard Module Structure

- Every major module should follow this structure:

  1. Learning Objectives
  2. Introduction
  3. Why This Feature Exists
  4. Mental Model
  5. Behind the Scenes
  6. Interface Walkthrough
  7. Every Option Explained
  8. Professional Workflow
  9. Real-World Examples
  10. Bug Bounty Use Cases
  11. Penetration Testing Use Cases
  12. Field Notes
  13. Common Mistakes
  14. Practice Exercises
  15. Interview Questions
  16. Quick Revision
  17. Self Assessment
  18. Key Takeaways
  19. Next Module

---

## Writing Style

* Use active voice.
* Keep paragraphs concise.
* Explain *why* before *how*.
* Introduce one concept at a time.
* Avoid unnecessary jargon.
* Define technical terms when first introduced.

---

## Markdown Standards

* Use `#` for the page title.
* Use `##` for major sections.
* Use `###` only when necessary.
* Prefer bullet points over long paragraphs.
* Use fenced code blocks with language identifiers where appropriate.
* Keep diagrams in plain text unless an image significantly improves understanding.

---

## Knowledge Boxes

- Use GitHub callouts for emphasis.

- Examples:

  * **NOTE** for useful observations.
  * **TIP** for productivity advice.
  * **IMPORTANT** for essential concepts.
  * **WARNING** for common pitfalls.

- Avoid excessive callouts; use them only where they add value.

---

## Examples

- Whenever possible, include:

  * HTTP requests
  * HTTP responses
  * API examples
  * Realistic URLs
  * Sample payloads
  * Practical scenarios

- Examples should be safe, educational, and easy to follow.

---

## Professional Perspective

- Every module should answer:

  * Why does this feature exist?
  * When should it be used?
  * When should it NOT be used?
  * How do professionals use it?
  * What mistakes do beginners make?

---

## Exercises

- Exercises should progress from:

  1. Observation
  2. Modification
  3. Analysis
  4. Investigation

- Encourage exploration rather than simply following instructions.

---

## Interview Preparation

- Include questions that test:

  * Conceptual understanding
  * Practical knowledge
  * Scenario-based reasoning

- Avoid trivia that has little practical value.

---

## Goal

- The purpose of this repository is not merely to document Burp Suite features.

- Its goal is to help readers develop the knowledge, confidence, and practical skills needed to use Burp Suite effectively during authorized web application security assessments.
