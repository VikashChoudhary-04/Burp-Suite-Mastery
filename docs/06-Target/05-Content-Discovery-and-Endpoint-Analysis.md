# Content Discovery and Endpoint Analysis

## Overview

- **Content discovery** is the process of identifying application resources that are relevant to understanding the authorized target.

- **Endpoint analysis** goes further by determining what those resources do, how they are reached, what inputs they accept, which identities can access them, and where they fit into the application's security model.

- Burp Suite's Target tool helps organize discovered content so that testing can move from:

  ```text
  "I found an endpoint"

  to

  "I understand what this endpoint does, who should access it,
  what data it handles, and why it matters."
  ```

- Discovery should support structured investigation rather than uncontrolled enumeration.

---

## Learning Objectives

- By the end of this lesson, you should be able to:

  * Explain the difference between content discovery and endpoint analysis.
  * Identify resources from normal application traffic.
  * Classify endpoints by function and security relevance.
  * Analyze paths, methods, parameters, and response behavior.
  * Recognize API and JavaScript-derived endpoint leads.
  * Compare authenticated, unauthenticated, and role-specific content.
  * Identify potentially important unlinked or conditional functionality.
  * Build a structured endpoint inventory.
  * Turn endpoint observations into testable security hypotheses.

---

## What Is Content Discovery?

- Content discovery identifies resources associated with an application.

- Examples include:

  * Pages.
  * Directories.
  * API routes.
  * JavaScript files.
  * Static resources.
  * Upload and download endpoints.
  * Administrative interfaces.
  * Versioned APIs.
  * GraphQL endpoints.
  * WebSocket endpoints.

- Discovery can occur naturally while browsing or through explicitly permitted discovery techniques.

---

## Passive and Active Discovery

### Passive Discovery

- Passive discovery relies primarily on application behavior already being observed.

- Examples:

  * Browsing application features.
  * Reviewing Proxy HTTP history.
  * Reviewing the Target Site Map.
  * Inspecting links and forms.
  * Observing API calls generated by the browser.
  * Reviewing JavaScript references.

### Active Discovery

- Active discovery sends additional requests to identify resources.

- Examples may include:

  * Authorized crawling.
  * Directory or file enumeration.
  * Endpoint probing.
  * API route enumeration.

- Active techniques must comply with:

  * Scope.
  * Rules of engagement.
  * Automation restrictions.
  * Rate limits.
  * Production-impact restrictions.

---

## Discovery Is Not Validation

- Finding a path does not prove that it exists, is reachable, is vulnerable, or is authorized for testing.

  ```text
  Endpoint Reference
        │
        ▼
  Verify Scope
        │
        ▼
  Verify Reachability
        │
        ▼
  Understand Function
        │
        ▼
  Establish Baseline
        │
        ▼
  Form Hypothesis
        │
        ▼
  Controlled Testing
        │
        ▼
  Validate Evidence
  ```

## Sources of Endpoint Information

- Endpoint leads may come from:

  * Site Map.
  * Proxy history.
  * HTML links.
  * Forms.
  * Redirects.
  * JavaScript files.
  * API responses.
  * Error messages.
  * Browser network activity.
  * Documentation exposed by the application.
  * Authorized crawling or discovery.
  * WebSocket messages.

- Each source provides a different level of confidence.

---

## Endpoint Inventory

- Maintain an inventory such as:

  ```text
  Method | Path | Function | Auth | Role | Inputs | Notes
  -------|------|----------|------|------|--------|------
  GET    | /profile | View profile | Yes | User | None | Sensitive data
  PATCH  | /api/profile | Update profile | Yes | User | JSON | State changing
  GET    | /orders/{id} | View order | Yes | User | ID | Ownership relevant
  POST   | /admin/users | Create user | Yes | Admin | JSON | Privileged
  ```

- The purpose is to capture security context, not just URLs.

---

## Analyze the HTTP Method

- The method often reveals the intended operation.

  ```text
  GET     → Retrieve
  POST    → Create / submit / trigger
  PUT     → Replace or update
  PATCH   → Partial update
  DELETE  → Delete
  ```

- These are conventions, not guarantees.

- Always observe actual application behavior.

---

## Same Path, Different Operations

- One path may support multiple methods.

  ```text
  GET /api/projects/42
  PATCH /api/projects/42
  DELETE /api/projects/42
  ```

- Treat each method as a distinct operation in your endpoint model.

- Security requirements may differ between reading, modifying, and deleting the same object.

---

## Path Analysis

- Paths often reveal application structure.

  ```text
  /users/42
  /orders/1001
  /projects/7/members
  /admin/settings
  /api/v2/accounts/88
  ```

- Look for:

  * Resource names.
  * Object identifiers.
  * Nested relationships.
  * Version numbers.
  * Privileged namespaces.
  * Action-oriented routes.

---

## Nested Resources

- Example:

  ```text
  /organizations/12/projects/55/members/8
  ```

- This suggests relationships:

  ```text
  Organization 12
       │
       └── Project 55
              │
              └── Member 8
  ```

- Such relationships may indicate multiple authorization boundaries.

---

## Action Endpoints

- Some endpoints represent actions rather than simple resources.

  ```text
  /orders/1001/cancel
  /users/42/disable
  /documents/8/share
  /account/change-email
  ```

- Record:

  * Who can trigger the action.
  * Required state.
  * Object ownership.
  * Whether the action is reversible.
  * Security impact.

---

## Query Parameters

- Example:

  ```text
  GET /search?q=phone&page=2
  ```

- Parameters may control:

  * Search terms.
  * Filtering.
  * Sorting.
  * Pagination.
  * Object selection.
  * Redirect destinations.
  * Display behavior.

- Record parameters that influence security-sensitive behavior.

---

## Body Parameters

- Example:

  ```http
  PATCH /api/profile HTTP/1.1
  Content-Type: application/json

  {
    "displayName": "Alice",
    "email": "alice@example.test"
  }
  ```

- Ask:

  * Which fields are user controlled?
  * Which fields affect identity or privilege?
  * Which values are generated by the client?
  * Which fields appear optional?
  * Does the request modify state?

---

## Headers and Cookies

- Security-relevant values may appear in:

  * Authorization headers.
  * Session cookies.
  * CSRF tokens.
  * Tenant headers.
  * API-version headers.
  * Custom role or context headers.

- Do not assume a header is trusted simply because the client sends it.

- Record its apparent purpose before testing.

---

## Response Analysis

- Responses help determine what an endpoint actually does.

- Review:

  * Status code.
  * Response body.
  * Content type.
  * Redirects.
  * Error messages.
  * Response length.
  * Caching behavior.
  * Security-relevant headers.
  * Data returned.

---

## Status Codes as Signals

- Examples:

  ```text
  200 → Request processed successfully
  201 → Resource created
  204 → Successful response without body
  301/302 → Redirect
  400 → Invalid request
  401 → Authentication required or invalid
  403 → Access denied
  404 → Resource not found or intentionally hidden
  405 → Method not allowed
  500 → Server-side error
  ```

- Status codes are useful signals but should not be interpreted without the response body and application context.

---

## Compare Response Behavior

- Establish a baseline.

  ```text
  Valid Request
      │
      ▼
  Record Status + Body + Length + Behavior
      │
      ▼
  Change One Relevant Variable
      │
      ▼
  Compare
  ```

- Differences may reveal:

  * Authentication behavior.
  * Authorization behavior.
  * Object existence.
  * Input validation.
  * Workflow state.

---

## Authentication Classification

- Classify endpoints:

  ```text
  Public

  Authentication Required

  Role Restricted

  State Restricted

  Unknown
  ```

- This makes security boundaries easier to visualize.

---

## Role-Specific Endpoint Analysis

- If authorized roles are available, compare them.

  ```text
  User:
  GET /dashboard
  GET /orders

  Manager:
  GET /dashboard
  GET /orders
  GET /reports

  Admin:
  GET /dashboard
  GET /users
  POST /users
  GET /settings
  ```

- Differences help identify intended privilege boundaries.

---

## API Endpoint Discovery

- APIs may appear through:

  * Browser XHR/fetch requests.
  * JavaScript.
  * Mobile-oriented routes.
  * API documentation.
  * GraphQL requests.
  * Error messages.

- Common patterns:

  ```text
  /api/
  /api/v1/
  /api/v2/
  /graphql
  /rest/
  ```

- Verify scope before testing separately hosted API infrastructure.

---

## API Versioning

- Multiple versions may coexist.

  ```text
  /api/v1/users
  /api/v2/users
  ```

- Record each version separately.

- Different versions may have different:

  * Authentication.
  * Input formats.
  * Features.
  * Authorization behavior.
  * Deprecation status.

---

## JavaScript-Derived Endpoints

- JavaScript can contain strings or code referencing routes.

  ```text
  /api/account/preferences
  /api/reports/export
  /feature-flags
  ```

- Treat these as discovery leads.

  ```text
  JavaScript Reference
        │
        ▼
  Confirm Scope
        │
        ▼
  Determine Whether Route Is Reachable
        │
        ▼
  Understand Expected Method and Context
  ```

- Do not treat a string in JavaScript as proof of a vulnerability.

---

## Unlinked Functionality

- Some valid functionality may not be linked from the current interface.

- Reasons include:

  * Role restrictions.
  * Feature flags.
  * Legacy functionality.
  * Conditional workflows.
  * Hidden navigation.
  * API-only behavior.

- If discovered, verify authorization and application context before interacting with it.

---

- Content types help classify endpoints.

  ```text
  text/html
  application/json
  application/xml
  text/javascript
  multipart/form-data
  application/pdf
  ```

- They can indicate:

  * Browser pages.
  * APIs.
  * Uploads.
  * Downloads.
  * JavaScript.
  * Documents.

## File and Download Endpoints

- Examples:

  ```text
  /files/42
  /documents/42/download
  /reports/export
  ```

- Record:

  * Object identifier.
  * Authentication requirement.
  * Ownership relationship.
  * Content type.
  * Filename behavior.
  * Whether sensitive data is returned.

---

## Upload Endpoints

- Examples:

  ```text
  POST /upload
  POST /api/documents
  ```

- Record:

  * Accepted content type.
  * Form fields.
  * Object ownership.
  * Authentication.
  * Resulting resource location.
  * Processing behavior.

---

## Administrative Endpoints

- Common patterns include:

  ```text
  /admin
  /management
  /staff
  /internal
  /api/admin
  ```

- Names alone do not prove privilege.

- Verify expected role and behavior.

---

## Error-Driven Discovery

- Error responses may reveal:

  * Route names.
  * Parameter names.
  * Expected methods.
  * Object types.
  * API versions.

- Record useful information without intentionally causing disruptive behavior.

---

## Redirect Analysis

- Redirects may reveal:

  * Authentication boundaries.
  * Canonical paths.
  * External identity providers.
  * Workflow transitions.

  ```text
  /account
      │
      ▼
  302 /login
  ```

- Or:

  ```text
  /login/sso
      │
      ▼
  identity-provider.example
  ```

- Verify third-party scope before testing redirected systems.

---

## Endpoint Classification Model

- Classify each important endpoint using:

  ```text
  Function
  +
  Method
  +
  Authentication
  +
  Role
  +
  Object
  +
  Inputs
  +
  State Change
  +
  Data Sensitivity
  ```

- Example:

  ```text
  PATCH /api/users/42

  Function:
  Update user

  Authentication:
  Required

  Role:
  Administrator expected

  Object:
  User 42

  Inputs:
  JSON fields

  State Change:
  Yes

  Sensitivity:
  High
  ```

## Example Analysis

- Observed:

  ```text
  GET /api/orders/1042
  ```

- Analysis:

  ```text
  Function:
  Retrieve order

  Authentication:
  Required

  Object:
  Order 1042

  Identifier:
  Numeric

  Data:
  Customer and order information

  Security Boundary:
  Object ownership

  Hypothesis:
  The server should verify that the authenticated identity
  is permitted to access order 1042.
  ```

- This is a hypothesis, not a vulnerability.

## Related Endpoint Families

- Group related routes.

  ```text
  Orders

  GET    /api/orders
  POST   /api/orders
  GET    /api/orders/{id}
  PATCH  /api/orders/{id}
  DELETE /api/orders/{id}
  POST   /api/orders/{id}/cancel
  ```

- Endpoint families reveal the complete object lifecycle.

---

## CRUD Mapping

- A useful model is:

  ```text
  Create
  Read
  Update
  Delete
  ```

- Map which operations exist and who should be permitted to perform each.

  ```text
  Resource: Project

  Create → Manager
  Read   → Member
  Update → Manager
  Delete → Owner
  ```

## Common Mistakes

- Treating endpoint discovery as a race to collect the largest URL list.

- Ignoring HTTP methods.

- Ignoring request bodies.

- Ignoring headers and cookies.

- Failing to map endpoint families.

- Assuming a `404` always means a resource does not exist.

- Assuming a `200` means authorization is correct.

- Treating JavaScript strings as confirmed live endpoints.

- Ignoring role-specific functionality.

- Performing active discovery without checking program restrictions.

- Reporting interesting behavior before validating impact.

---

## Professional Tips

- Begin with normal application usage.

- Record endpoint purpose, not just path.

- Group endpoints by resource and workflow.

- Track authentication and role requirements.

- Mark state-changing operations.

- Record identifiers and object relationships.

- Compare browser routes with APIs.

- Use response behavior to establish baselines.

- Keep discovery evidence separate from vulnerability evidence.

- Revisit endpoint coverage after changing roles or application state.

## Key Takeaways

- Content discovery identifies application resources; endpoint analysis explains their security context.

- Useful endpoint analysis includes methods, inputs, roles, objects, state changes, and response behavior.

- Normal application traffic, Site Map, Proxy history, APIs, and JavaScript are valuable discovery sources.

- Related endpoints should be grouped into resource families and workflows.

- Authenticated, unauthenticated, and role-specific functionality should be mapped separately.

- Active discovery must remain within authorization and engagement restrictions.

- Endpoint observations should lead to structured hypotheses, followed by controlled testing and evidence-based validation.
