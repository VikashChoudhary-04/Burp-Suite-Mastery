# HTTP Security Reasoning for Burp Users

> **Module Progress**

```text
████████████░ 12 / 13 Lessons
```

- Burp Suite gives you the ability to:

  * Intercept requests.
  * Modify parameters.
  * Replay traffic.
  * Compare responses.
  * Inspect sessions.
  * Observe application behavior.

- But modifying HTTP traffic is not the same as performing effective security testing.

- The real skill is reasoning.

- A strong tester moves through:

  ```text
  Observation

  ↓
  
  Understanding

  ↓

  Hypothesis
  
  ↓

  Controlled Test

  ↓

  Evidence

  ↓

  Verification

  ↓

  Conclusion
  ```  

- This lesson brings the previous HTTP concepts together into a repeatable security-testing methodology.

---

## The Core Testing Model

- A useful workflow is:

  ```text
  Baseline

  ↓  

  Map Trust and State

  ↓

  Form Hypothesis

  ↓
  
  Change One Relevant Variable

  ↓

  Observe Differences

  ↓

  Verify Actual Impact

  ↓

  Document Evidence
  ```

- This is stronger than randomly modifying requests until something unusual happens.

---

## Start With the Baseline

- Before changing anything, understand normal behavior.

- Capture:

  ```text
  Request

  Response

  Authentication State
  
  Object Ownership

  Workflow State

  Expected Result
  ```

- Example:

  ```http
  GET /api/orders/4821 HTTP/1.1
  Host: example.com
  Cookie: session=USER_A
  ```

- Normal response:

  ```http
  HTTP/1.1 200 OK
  Content-Type: application/json

  {
    "id": 4821,
    "owner": "USER_A"
  }
  ```

- This becomes your baseline.

---

## Why Baselines Matter

- Suppose you change:

  ```text
  4821
  ```

- to:

  ```text
  4822
  ```

- and receive:

  ```text
  200 OK
  ```  

- Without a baseline, you still do not know:

  * Who owns `4822`.
  * Whether the data is public.
  * Whether the response actually contains another user's data.
  * Whether authorization should have prevented access.

- A response difference only becomes meaningful when context is understood.

---

## Trust Boundaries

- A trust boundary exists where data or authority crosses between components with different trust levels.

- Common web application boundaries include:

  ```text
  User

  ↓

  Browser

  ↓

  Reverse Proxy / CDN

  ↓

  Application
  
  ↓

  Database / Internal Services
  ```  

- At each boundary, ask:

  > What information is being trusted?

---

## Client-Controlled Data

- Potentially client-controlled inputs include:

  ```text
  Path

  Query Parameters

  Request Body

  Headers

  Cookies

  File Metadata

  Hidden Fields
  ```  

- Even if the normal browser generates a value automatically, a user controlling the HTTP request may often modify it.

- Therefore:

  ```text
  Generated by Browser

  ≠

  Trusted by Server
  ```  

---

## Server-Controlled Data

- Examples may include:

  ```text
  Database Records

  Server-Side Session State

  Authorization Policies

  Server-Calculated Prices

  Internal Object Ownership

  Cryptographically Verified State
  ```

- A secure design should base critical decisions on authoritative information rather than blindly trusting client claims.

---

## Trust Must Be Verified

- Suppose a request contains:

  ```json
  {
    "productId": 4821,
    "price": 1000
  }
  ```

- Ask:

  ```text
  Who supplied price=1000?

  ↓

  Client
  ```  

- Then:

  ```text
  Does the server trust it?

  or

  Does the server calculate the price independently?
  ```  

- The request alone cannot answer this.

- You need controlled testing.

---

## Security-Relevant Input Categories

- Classify inputs by what they may influence.

### Identity

```text
userId

accountId

email

username
```

### Authorization

```text
role

permission

tenantId

ownerId
```

### Object Selection

```text
orderId

invoiceId

fileId
```

### Business Logic

```text
price

quantity

discount

status
```

### Navigation / Destination

```text
redirect

next

returnUrl
```

### Server-Side Resource Selection

```text
file

path

template

url
```

- Classification helps prioritize meaningful hypotheses.

---

## Identity, Object, Action

- For every security-sensitive request, identify three things:

  ```text
  Identity

  Object

  Action
  ```  

- Example:

  ```http
  DELETE /api/orders/4821 HTTP/1.1
  Cookie: session=USER_A
  ```

- Map it:

  ```text
  Identity:
  USER_A

  Object:
  Order 4821

  Action:
  Delete
  ```

- Then ask:

  > Is this identity authorized to perform this action on this object?

- This simple model is extremely powerful.

---

## Authorization Matrix

- You can represent expected access:

| Identity | Object         | Action                | Expected                      |
| -------- | -------------- | --------------------- | ----------------------------- |
| User A   | User A's order | Read                  | Allow                         |
| User A   | User B's order | Read                  | Deny unless explicitly shared |
| User A   | User B's order | Delete                | Deny                          |
| Admin    | User B's order | Administrative action | Depends on policy             |

- This makes authorization assumptions explicit.

---

## Authentication Is Not Authorization

- A server may correctly verify:

  ```text
  Session = Valid User A
  ```

- and still fail to verify:

  ```text
  Does User A Own Order 4822?
  ```

- Therefore:

  ```text
  Authenticated

  ≠

  Authorized for Every Resource
  ```  

- Always separate the two questions.

---

## Validation Is Not Authorization

- Suppose:

  ```json  
  {
    "accountId": 105
  }
  ```

- The server may validate:

  ```text
  Is accountId an integer?

  Yes.
  ```

- But security requires another question:

  ```text
  Is the authenticated user allowed to act on account 105?
  ```

- Therefore:

  ```text
  Valid Input

  ≠

  Authorized Input
  ```

---

## Authentication vs Authorization vs Validation

- Keep these distinct:

  ```text
  Authentication

  Who are you?
  ```

  ```text
  Authorization

  Are you allowed to do this?
  ```  

  ```text
  Validation

  Is this input acceptable in format/value/context?
  ```

- A request may pass one control and fail another.

---

## Source and Sink Mental Model

- A useful security concept is:

  ```text
  Source

  ↓

  Data Flow

  ↓

  Sink
  ```  
  
- A **source** is where potentially untrusted data enters.

- A **sink** is where that data is used in a security-sensitive operation or interpreter.

---

## Example

- Request:

  ```text
  ?search=alice
  ```

- Potential source:

  ```text
  search parameter
  ```

- Possible sinks might include:

  ```text
  Database Query

  HTML Output
  
  Operating System Command

  File Path

  Server-Side HTTP Request

  Template Engine
  ```  

- The parameter's existence does not prove a vulnerability.

- The question is:

  > Where does the data flow, and how is it handled there?

---

## Source Does Not Equal Vulnerability

- Suppose:

  ```text
  ?name=Alice
  ```

is user-controlled.

- That only establishes:

  ```text
  Untrusted Input Exists
  ```

- A vulnerability requires additional evidence about:

  ```text
  Where It Goes

  How It Is Processed

  What Security Control Exists

  What Impact Is Possible
  ```  

- Do not jump from input discovery directly to vulnerability claims.

---

## Build a Hypothesis

- A good hypothesis is specific and testable.

- Weak:

  ```text
  Maybe this endpoint has IDOR.
  ```

- Better:

  ```text
  The order ID appears client-controlled.

  I hypothesize that the server may retrieve orders
  using this identifier without verifying ownership.
  ```

- Now the test has a purpose.

---

## Hypothesis Structure

- Use:

  ```text
  Observation

  ↓

  Security Expectation

  ↓

  Possible Failure

  ↓

  Test
  ```  

- Example:

  ```text
  Observation:
  Request contains /orders/4821.

  Security Expectation:
  Only the owner should access this order.

  Possible Failure:
  Server may check authentication but not ownership.

  Test:
  Use controlled User A session to request
  a known User B order.
  ```

---

## One Variable at a Time

- Baseline:

  ```http
  GET /api/orders/4821 HTTP/1.1
  Cookie: session=USER_A
  ```

- Test:

  ```http
  GET /api/orders/4822 HTTP/1.1
  Cookie: session=USER_A
  ```

- Only:

  ```text
  Order ID
  ```

changed.

- This gives stronger causal evidence.

---

## Why Multiple Changes Are Weak

- Suppose you simultaneously change:

  ```text
  Session Cookie

  Order ID

  Role Header

  HTTP Method
  ```  

and the response changes.

- Which modification caused it?

  - You do not know.

- Controlled testing reduces ambiguity.

---

## Controlled Mutation Workflow

- Use:

  ```text
  Baseline Request

  ↓

  Duplicate Request

  ↓

  Change One Relevant Element
  
  ↓

  Send

  ↓

  Compare

  ↓

  Restore
  
  ↓

  Test Next Hypothesis
  ```

- This is the foundation of effective Repeater usage.

---

## Compare More Than Status Codes

- Compare:

  ```text
  Status Code

  Response Length

  Headers

  Body

  Cookies
    
  Redirects
  
  Timing
  
  Application State
  ```  

- A security-relevant difference may appear anywhere.

---

## Example

- Baseline:

  ```text
  403 Forbidden
  Length: 120
  ```

- Modified:

  ```text
  403 Forbidden
  Length: 8,420
  ```

- Same status.

- Very different response.

- Inspect the body.

---

## Negative Controls

- A negative control is a request expected **not** to produce the suspected behavior.

- Suppose you believe User A can access User B's order.

- Useful comparisons might include:

  ```text
  User A → Own Order

  User A → User B Order

  Unauthenticated → User B Order

  Invalid Order ID
  ```

- These comparisons help determine what condition actually controls access.

---

## Positive Controls

- A positive control is a known-valid case.

- Example:

  ```text
  User B Session

  ↓

  User B Order

  ↓

  Expected 200
  ```  

- This confirms that:

  * The object exists.
  * The endpoint works.
  * The test data is valid.

- Without this, a `404` under User A may be difficult to interpret.

---

## Two-User Testing Model

- Use controlled accounts:

  ```text
  User A

  Owns Object A
  ```

  ```text
  User B

  Owns Object B
  ```

- Then compare:

  ```text
  A Session + A Object
  ```

  ```text
  B Session + B Object
  ```

  ```text
  A Session + B Object
  ```

  ```text
  B Session + A Object
  ```

- This creates strong authorization evidence.

---

## Expected vs Observed Behavior

- Document both.

- Example:

  ```text
  Expected:

  User A should receive access denied  
  when requesting User B's private invoice.
  ```

  ```text
  Observed:

  User A received HTTP 200 and the complete
  invoice belonging to User B.
  ```

- The difference between expectation and observation defines the issue.

---

## Verify Ownership

- Do not claim:

  ```text
  I changed ID 105 to 106,
  therefore I accessed another user's object.
  ```

- First prove:

  ```text
  Who owns 105?

  Who owns 106?
  ```

- Use controlled accounts or reliable application evidence.

- Object enumeration alone is not proof of broken authorization.

---

## Verify Actual Impact

- Suppose you modify:

  ```json  
  {
    "role": "admin"
  }
  ```

- and receive:

  ```text
  200 OK
  ```  

- That does not prove privilege escalation.

- Verify:

  ```text
  Did the role actually change?

  Can the account perform admin actions?

  Does a fresh session reflect the new privilege?
  ```  

- Impact requires observable evidence.

---

## Response Success Is Not State Success

- Remember:

  ```text
  200

  ≠

  Action Definitely Succeeded
  ```

  and:

  ```text
  500

  ≠

  Action Definitely Failed
  ```

- Verify state independently.

---

## Error Messages Are Evidence, Not Conclusions

- Suppose:

  ```http
  HTTP/1.1 500 Internal Server Error

  SQL syntax error near...
  ```

- This may suggest:
  
  ```text
  User Input Reached a Database-Related Code Path
  ```

- But it does not automatically prove:

  ```text
  Exploitable SQL Injection
  ```

- Additional controlled evidence is required.

---

## Avoid Overclaiming

- Weak conclusion:

  ```text
  500 means SQL injection.
  ```

- Better:

  ```text
  The modified input consistently triggers a database
  syntax error that is absent from the baseline.

  This indicates the input may influence database query
  construction and warrants further authorized validation.
  ```

- Claims should match evidence.

---

## Evidence Strength

- Think of evidence in levels.

### Weak

```text
Interesting parameter name
```

### Better

```text
Changing parameter changes response
```

### Stronger

```text
Controlled change produces predictable security-relevant behavior
```

### Strong

```text
Behavior is reproduced and actual impact is verified
```

- Professional reporting should rely on the strongest evidence available.

---

## Reproducibility

- A finding should ideally be reproducible.

- Record:

  ```text
  Prerequisites

  Account / Role

  Starting State

  Exact Request

  Exact Modification

  Observed Response

  Verification Step
  ```  

- If you cannot reproduce the behavior, investigate why before making a strong claim.

---

## False Positives

- Common causes include:

  ```text
  Caching

  Stale Sessions

  Expired Tokens

  Race / Timing
  
  Dynamic Content

  Load Balancing

  Different Test Data
  
  Workflow State

  Rate Limiting
  ```

- Unexpected behavior should be repeated under controlled conditions.

---

## Caching Example

- You modify:

  ```text
  ?id=106
  ```  

but receive content that appears unchanged.

- Possible explanations:

  ```text
  Parameter Ignored

  Cache Returned Old Response

  Server Normalized Input

  Object Data Is Identical
  ```

- Do not assume immediately.

- Investigate.

---

## Timing Evidence

- Suppose a request takes:

  ```text
  Baseline:
  200 ms
  ```

- Modified:

  ```text
  5 seconds
  ```

- One slow response proves little.

- Repeat controlled tests and consider:

  ```text
  Network Variation

  Server Load

  Rate Limiting

  Backend Processing

  Intentional Delay
  ```  

- Timing requires careful comparison.

---

## Security Control Identification

- When analyzing behavior, identify which control should protect it.

- Examples:

  ```text
  Authentication

  Authorization

  Input Validation

  CSRF Protection

  Rate Limiting

  Session Management
  
  Business Logic Validation

  Output Encoding
  ```

- Then test whether that specific control behaves correctly.

---

## Defense Layers

- A request may pass through multiple controls:

  ```text
  Authentication
  
  ↓

  Authorization

  ↓

  Input Validation

  ↓

  Business Logic

  ↓

  Database / Backend Operation
  
  ↓

  Output Encoding
  ```

- A failure at one layer does not automatically imply failure at every other layer.

---

## Client-Side vs Server-Side Controls

- Suppose JavaScript prevents:

  ```text
  quantity > 10
  ```

- The security question is:

  ```text
  What happens if the HTTP request contains quantity=100?
  ```

- Client-side restrictions are not sufficient for security-sensitive enforcement.

---

## Server Rejection Is Not Always Security

- Suppose modified input returns:

  ```text
  400 Bad Request
  ```

- Possible explanations:

  ```text
  Input Validation

  Parser Error

  Missing Dependency

  Security Filter

  Malformed Syntax
  ```

- Determine why the request failed before claiming a defense worked.

---

## Authorization Testing Example

- Baseline:

  ```http
  GET /api/profile/105 HTTP/1.1
  Cookie: session=USER_A
  ```

- Known:

  ```text
  Profile 105 belongs to User A.
  ```

- Controlled test:

  ```http
  GET /api/profile/106 HTTP/1.1  
  Cookie: session=USER_A
  ```

- Known:

  ```text
  Profile 106 belongs to controlled User B.
  ```

- Expected:

  ```text
  Access denied or appropriately restricted.
  ```

- Observed:

  ```text
  200 OK

  User B private profile returned.
  ```

- Verification:

  ```text
  User B confirms profile 106 contains the same data.
  ```
  
- This is strong evidence of broken object-level authorization.

---

## Business Logic Example

- Normal checkout request:

  ```json
  {
    "productId": 4821,
    "quantity": 1,
    "price": 1000
  }
  ```

- Hypothesis:

  ```text
  The server may trust the client-supplied price.
  ```

- Test:

  ```json
  {
    "productId": 4821,
    "quantity": 1,
    "price": 1
  }
  ```

- Do not conclude from:

  ```text
  200 OK
  ```

- Verify:

  ```text
  What amount appears in the order?

  What amount is actually charged?

  Does the server recalculate later?
  ```

- Impact exists only if the authoritative transaction is affected.

---

## Authentication Example

- Observation:

  ```text
  POST /login

  ↓

  302 /mfa
  ```

- Hypothesis:

  ```text
  The session may already have full privileges
  before MFA completion.
  ```

- Test:

  ```text
  Use the intermediate session to request
  a protected endpoint directly.
  ```

- Verify:

  ```text
  Does protected data or functionality become accessible?
  ```

- The redirect itself is not the security control.

---

## Parameter Testing Example

- Request:

  ```http
  GET /search?q=alice HTTP/1.1
  ```

- Do not immediately test every payload you know.

- First ask:

  ```text
  What does q appear to control?

  Where is its value reflected?

  Does it affect database-backed results?

  Is output encoded?

  Does behavior change predictably?
  ```

- Understand the data flow before choosing tests.

---

## Investigation Journal

- For each hypothesis, record:

  ```text
  Observation:

  Hypothesis:

  Baseline:

  Variable Changed:

  Expected Result:

  Observed Result:

  Verification:

  Conclusion:

  Next Test:
  ```

- This creates disciplined reasoning and makes later reporting easier.

---

## From Observation to Finding

- A useful progression:

  ```text
  Observation

  ↓

  Potential Security Relevance

  ↓

  Hypothesis

  ↓

  Controlled Validation

  ↓

  Verified Impact

  ↓

  Finding
  ```

- Not every observation becomes a vulnerability.

- That is normal.

---

## Observation Example

```text
The request contains:
X-Role: user
```

- This is only an observation.

---

## Hypothesis

```text
The application may trust X-Role
for authorization decisions.
```

- Now you have something testable.

---

## Controlled Test

- Modify:

  ```text
  X-Role: admin
  ```

while keeping other variables constant.

- Observe behavior.

---

## Possible Outcome 1

- Nothing changes.

- Conclusion:

  ```text
  No evidence that this header controls authorization.
  ```

- Do not report a vulnerability.

---

## Possible Outcome 2

- Admin navigation appears in the response.

- Still verify.

- Ask:

  ```text
  Can an admin-only server-side action actually be performed?
  ```

- UI differences alone may not prove authorization bypass.

---

## Possible Outcome 3

- A normal user successfully performs a confirmed admin-only action.

- Now the evidence is much stronger.

---

## Severity Requires Impact

- A technically unusual behavior is not automatically high severity.

- Consider:

  ```text
  What data is exposed?

  What action becomes possible?

  Which users are affected?

  What privileges are required?

  How reproducible is it?

  What business consequence exists?
  ```  

- Severity should reflect demonstrated impact and realistic exploitability.

---

## Do Not Inflate Findings

- Avoid reporting:

  ```text
  Critical vulnerability
  ```

when evidence only shows:

  ```text
  Unexpected response
  ```

- Likewise, do not understate a clearly demonstrated cross-account or privilege-boundary failure.

- Evidence should drive severity.

---

## Safe Testing

- Professional testing requires:

  ```text
  Defined Scope

  Explicit Authorization

  Controlled Accounts

  Minimal Necessary Impact

  Careful Data Handling

  Reversible Actions Where Possible
  ```  

- Do not access unrelated real-user data merely to prove a point when safer validation is available.

---

## Stop Conditions

- Stop or reassess when testing risks:

  ```text
  Service Disruption

  Data Destruction

  Real Financial Transactions

  Unnecessary Sensitive Data Access

  Mass Account Impact

  Scope Violation
  ```  

- A technically possible test is not automatically an appropriate test.

---

## Reporting Evidence

- A strong finding usually contains:

  ```text
  Title

  Affected Endpoint

  Prerequisites

  Steps to Reproduce

  Expected Behavior
  
  Observed Behavior
  
  Security Impact

  Evidence

  Remediation Guidance
  ```

- HTTP evidence should support the reasoning clearly.

---

## Good Evidence

- Include only what is necessary:

  ```text
  Relevant Request

  Relevant Response

  Changed Parameter

  Controlled Account Context

  Verification Result
  ```  

- Redact:

  ```text
  Session Tokens

  Passwords

  Sensitive Personal Data

  Unnecessary Secrets
  ```  

---

## Evidence vs Assumption

- Label your reasoning correctly.

### Evidence

```text
User A session returned User B's controlled invoice.
```

### Inference

```text
The endpoint likely lacks object-level authorization.
```

### Assumption

```text
All invoices may be accessible.
```

- Do not present assumptions as verified facts.

- Use this complete workflow:

  ```text

  ↓

  ↓

  3. Identify Identity

  ↓

  4. Identify Object

  ↓

  5. Identify Action

  ↓

  6. Identify Client-Controlled Inputs

  ↓

  7. Identify Trust Boundaries

  ↓

  8. Identify Expected Security Control

  ↓

  9. Form Specific Hypothesis

  ↓

  10. Establish Positive / Negative Controls
  ```  

## Common Mistakes

- Avoid:

  ```text
  Interesting response = vulnerability.
  ```

  ```text
  200 = exploit succeeded.
  ```

  ```text
  403 = secure.
  ```

  ```text
  500 = SQL injection.
  ```

  ```text
  Changing an ID = IDOR.
  ```

  ```text
  Encoded value = protected value.
  ```

  ```text
  Hidden field = trusted field.
  ```

  ```text
  Authenticated = authorized.
  ```

  ```text
  Client-side restriction = security control.
  ```

  ```text
  One unusual response = reproducible finding.
  ```

  ```text
  Scanner output = verified vulnerability.
  ```  

- Security testing requires evidence and context.

## Final Investigation Checklist

```text
[ ] Establish a valid baseline.

[ ] Understand the complete request.

[ ] Understand the complete response.

[ ] Identify authentication state.

[ ] Identify identity, object, and action.

[ ] Identify client-controlled inputs.

[ ] Identify authoritative server-side state.

[ ] Identify trust boundaries.

[ ] Identify expected security controls.

[ ] Form a specific hypothesis.

[ ] Use controlled accounts where appropriate.

[ ] Establish positive and negative controls.

[ ] Change one relevant variable at a time.

[ ] Compare status, headers, body, length, timing, and state.

[ ] Verify ownership before claiming cross-user access.

[ ] Verify actual state before claiming action success.

[ ] Reproduce important behavior.

[ ] Distinguish evidence, inference, and assumption.

[ ] Avoid unnecessary impact.

[ ] Report only what the evidence supports.
```

---

## Key Takeaways

* Burp Suite enables testing, but reasoning determines test quality.
* Start with a baseline before modifying traffic.
* Identify trust boundaries and distinguish client-controlled from authoritative server-side data.
* Map identity, object, and action for security-sensitive requests.
* Authentication, authorization, and validation solve different problems.
* A source of untrusted input is not automatically a vulnerability.
* Form specific, testable hypotheses instead of performing random mutations.
* Change one relevant variable at a time.
* Use positive and negative controls to strengthen conclusions.
* Compare complete behavior, not only HTTP status codes.
* Verify object ownership and actual state before claiming impact.
* Error messages are evidence, not automatic vulnerability proof.
* Reproduce findings before reporting them.
* Distinguish evidence, inference, and assumptions.
* Severity should reflect demonstrated impact.
* Safe, scoped, minimally invasive testing is part of professional methodology.
* Report only conclusions supported by evidence.
