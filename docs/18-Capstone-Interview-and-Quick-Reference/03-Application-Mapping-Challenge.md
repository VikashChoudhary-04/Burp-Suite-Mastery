## Overview

* Before testing an application for vulnerabilities, you must understand how the application works.

* Application mapping is the process of systematically discovering and documenting:

  * Features.
  * Pages.
  * Endpoints.
  * HTTP methods.
  * Parameters.
  * APIs.
  * User roles.
  * Objects.
  * Object identifiers.
  * Authentication boundaries.
  * Authorization boundaries.
  * Tenant relationships.
  * State-changing operations.
  * Client-side functionality.
  * Integrations.
  * Hidden attack surfaces.

* The objective of this challenge is to transform an unfamiliar application from:

```text
Unknown Application
```

* Into:

```text
Structured Security Model
```

* A strong application map becomes the foundation for every later phase of the assessment.

```text
Browse

↓

Observe

↓

Inventory

↓

Connect

↓

Identify Trust Boundaries

↓

Prioritize Attack Surface

↓

Generate Hypotheses
```

# Common Mistakes

* Starting vulnerability scanning before understanding the application.
* Mapping only visible pages.
* Ignoring background API requests.
* Ignoring JavaScript.
* Failing to record HTTP methods.
* Recording endpoints without parameters.
* Failing to identify object IDs.
* Failing to track object ownership.
* Ignoring tenant relationships.
* Treating all authenticated users as equivalent.
* Failing to build a role-permission matrix.
* Testing only GET requests.
* Ignoring update and delete operations.
* Ignoring file functionality.
* Ignoring imports and exports.
* Ignoring server-side URL features.
* Ignoring integrations.
* Ignoring older API versions.
* Ignoring hidden routes.
* Failing to map business workflows.
* Failing to define business invariants.
* Treating every endpoint as equally important.
* Generating payloads before generating hypotheses.
* Relying entirely on automated crawling.
* Failing to update the map as new functionality is discovered.

---

# Professional Tips

* Browse normally before testing aggressively.
* Map one feature completely before moving randomly through the application.
* Keep Proxy history visible while exploring.
* Use Target Site map to understand structure.
* Record endpoints immediately.
* Record methods as well as paths.
* Track parameters and their locations.
* Highlight security-sensitive parameters.
* Record every meaningful object identifier.
* Create controlled objects to establish ownership.
* Label roles clearly.
* Map tenants early.
* Identify state-changing requests.
* Treat APIs as independent attack surfaces.
* Review JavaScript for hidden functionality.
* Search for alternate API versions.
* Identify file workflows.
* Identify URL-fetching functionality.
* Map complete business workflows.
* Write down business invariants.
* Look for sensitive values controlled by the client.
* Identify external trust boundaries.
* Observe error behavior.
* Prioritize identity, authorization, sensitive data, and critical business actions.
* Convert interesting observations into hypotheses.
* Keep mapping documents updated throughout the assessment.
* Do not confuse application mapping with vulnerability confirmation.

# Key Takeaways

* Application mapping transforms an unfamiliar target into a structured security model.
* Deep vulnerability testing should generally begin after the tester understands the application's major functionality and trust boundaries.
* Visible pages represent only part of the attack surface.
* Burp Proxy history reveals requests generated by normal application behavior.
* Target Site map helps organize hosts, directories, endpoints, parameters, APIs, and resources.
* Endpoint inventories should record methods, paths, authentication requirements, roles, parameters, objects, and state changes.
* HTTP methods matter because read, update, and delete operations may enforce authorization differently.
* Parameter inventories reveal what the client controls and which values may influence security-sensitive behavior.
* Object identifiers should be mapped together with ownership and tenant context.
* Role-permission matrices establish expected authorization behavior before access-control testing begins.
* Multi-tenant applications require explicit mapping of tenant membership and cross-tenant boundaries.
* Authentication boundaries distinguish public functionality from functionality requiring an authenticated identity.
* Authorization boundaries include horizontal, vertical, ownership-based, and tenant-based relationships.
* State-changing requests deserve special attention because they modify application data or workflow state.
* APIs should be mapped independently rather than treated merely as implementation details behind the user interface.
* JavaScript may reveal hidden endpoints, API versions, routes, parameters, WebSockets, GraphQL operations, feature flags, and integrations.
* Functionality not visible in the UI may still be directly reachable.
* File upload, download, import, export, preview, and processing functionality should be mapped explicitly.
* Features that accept URLs, callbacks, webhooks, or remote resources may expose server-side request attack surfaces.
* Business workflows should be mapped as sequences of states rather than isolated HTTP requests.
* Business invariants define rules that should remain true and provide the foundation for later business-logic testing.
* Sensitive client-controlled values such as roles, prices, ownership fields, permissions, and statuses deserve careful investigation.
* External integrations introduce additional trust boundaries.
* Error behavior may reveal object existence, internal architecture, framework details, or sensitive implementation information.
* Trust-boundary maps connect users, roles, tenants, application components, internal services, and third parties.
* Attack surfaces should be prioritized according to security impact rather than endpoint count.
* Strong application mapping naturally produces security hypotheses for later testing.
* Professional mapping progresses from **features → endpoints → methods → parameters → objects → roles → tenants → workflows → trust boundaries → attack-surface priorities → security hypotheses**.
