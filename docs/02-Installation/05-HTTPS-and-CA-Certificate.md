# HTTPS and CA Certificate

> **Module Progress**

```text id="r6w9mt"
█████░░░░ 5 / 9 Lessons
```

- Most modern web applications use HTTPS.

- For Burp Suite to inspect HTTPS traffic, the browser must trust certificates generated by Burp for the intercepted connections.

- Understanding why this is necessary is more important than simply memorizing certificate installation steps.

---

## HTTPS Without Burp

- Normally:

  ```text id="m4q8vn"
  Browser

  ↓

  TLS-Encrypted Connection

  ↓

  Web Application
  ```

- TLS helps provide:

  * Confidentiality.
  * Integrity.
  * Server authentication.

- The browser verifies the server's certificate before trusting the connection.

---

## What Changes When Burp Is Introduced?

- With interception:

  ```text id="p7n3kx"
  Browser

  ↓

  Burp Suite

  ↓

  Web Application
  ```

- Burp needs to inspect the HTTP messages inside the encrypted communication.

- Conceptually, Burp manages separate TLS relationships:

  ```text id="v5m9qw"
  Browser

  ⇅

  TLS

  ⇅

  Burp

  ⇅

  TLS

  ⇅

  Server
  ```

- This allows Burp to inspect and modify HTTP traffic during authorized testing.

---

## The Certificate Trust Problem

- Suppose the browser requests:

  ```text id="n8q2wr"
  https://example.com
  ```

- The browser expects a certificate it can validate for that connection.

- During interception, Burp presents a dynamically generated certificate appropriate to the requested host.

- The browser must decide:

  > Should I trust the authority that issued this certificate?

- Without Burp's CA being trusted, the browser may display a certificate warning.

---

## What Is a CA?

- CA means:

  > **Certificate Authority**

- A certificate authority signs certificates.

- Browsers maintain trusted CA stores.

- Conceptually:

  ```text id="k3p7mv"
  Trusted CA

  ↓

  Signs Certificate

  ↓

  Browser Verifies Signature

  ↓

  Connection Trusted
  ```

- Burp uses its own CA for intercepted TLS connections.

---

## Burp's CA Certificate

- Burp provides a CA certificate that can be trusted by a dedicated testing browser or profile.

- Once appropriately trusted:

  ```text id="x6m2pk"
  Burp CA

  ↓

  Trusted by Testing Browser

  ↓

  Burp-Generated Site Certificates

  ↓

  Accepted During Authorized Interception
  ```

- This allows HTTPS interception without repeated certificate warnings in compatible workflows.

---

## Important Security Meaning

- Installing a CA certificate is a significant trust decision.

- A trusted CA can influence which certificates the browser accepts.

- Therefore:

  > Install Burp's CA only in environments where that trust is intentionally required.

- Prefer a dedicated testing browser or profile.

---

## Avoid Your Everyday Browser Profile

- A strong setup is:

  ```text id="q9n4rv"
  Dedicated Testing Browser/Profile
  
  ↓

  Burp Proxy Configuration

  +

  Burp CA Trust
  ```

rather than modifying your normal everyday browsing environment unnecessarily.

- Isolation makes the testing environment easier to manage and safer to reset.

---

## Obtaining the Burp CA Certificate

- When the browser is correctly routed through Burp, Burp provides a local mechanism for obtaining its CA certificate.

- The exact interface or workflow can vary by Burp and browser version.

- Follow the current official Burp documentation for the precise installation procedure for your environment.

- The conceptual process is:

  ```text id="r4m8qn"
  Browser Routes Through Burp

  ↓

  Obtain Burp CA Certificate

  ↓

  Import Into Appropriate Trust Store

  ↓

  Trust for Website Identification as Required

  ↓

  Test HTTPS
  ```

---

## Browser Trust Stores Can Differ

- A common troubleshooting mistake is assuming every browser uses certificates identically.

- Depending on the operating system and browser, trust may come from:

  * Operating-system certificate stores.
  * Browser-specific certificate stores.
  * Managed enterprise policies.

- Therefore:

  ```text id="t7p2kx"
  Certificate Installed Somewhere

  ≠

  Certificate Trusted by This Browser
  ```

- Verify the trust location relevant to your actual testing browser.

---

## How HTTPS Interception Works Conceptually

- Suppose you browse:

  ```text id="m5q9vw"
  https://example.com
  ```

- A simplified model is:

  ```text id="p3n7kr"
  Browser Requests example.com

  ↓

  Traffic Reaches Burp

  ↓

  Burp Establishes Connection to example.com

  ↓

  Burp Presents an Interception Certificate to Browser

  ↓

  Browser Trusts Burp CA
  
  ↓

  Encrypted Sessions Established

  ↓

  Burp Can Inspect HTTP Messages
  ```

- Burp is not "breaking HTTPS."

- Your testing browser has explicitly been configured to trust Burp as part of the authorized interception environment.

---

## HTTP Works, HTTPS Fails

- This is one of the most useful diagnostic patterns.

- Suppose:

  ```text id="n6q2mv"
  HTTP Site

  ↓

  Loads Through Burp
  ```

- but:

  ```text id="k9p4rx"
  HTTPS Site

  ↓

  Certificate Warning / TLS Error
  ```

- This strongly suggests:

  ```text id="v2m8qw"
  Browser → Burp Connectivity

  Probably Working
  ```

- while:

  ```text id="q5n3pk"
  TLS Trust / HTTPS Configuration

  Needs Investigation  
  ```

- Do not unnecessarily rebuild the proxy listener first.

---

## Common HTTPS Problems

### Problem 1 — CA Not Installed

- Symptoms may include certificate trust warnings.

- Investigate whether the Burp CA has been imported into the correct testing browser trust store.

---

### Problem 2 — CA Installed in Wrong Store

- The certificate may exist on the machine but not be trusted by the browser actually being used.

- Verify the browser's trust model.

---

### Problem 3 — Wrong Browser Profile

- You may have installed the CA in:

  ```text id="r8m4nv"
  Profile A
  ```

- while testing with:

  ```text id="t3q7mx"
  Profile B
  ```

- Keep testing environments clearly separated.

---

### Problem 4 — Traffic Is Not Actually Going Through Burp

- If the browser is bypassing Burp, certificate troubleshooting may be misleading.

- Check:

  ```text id="p6n2kw"
  Proxy → HTTP History
  ```

- Does the request appear?

- If not, investigate browser routing first.

---

### Problem 5 — Application Uses Additional TLS Protections

- Some applications or clients may use mechanisms that make ordinary interception more difficult.

- Examples can include:

  * Certificate pinning.
  * Mutual TLS.
  * Specialized client trust behavior.

- Do not assume every TLS failure has the same cause.

---

## Certificate Pinning

- Some clients are designed to trust only specific certificates or public keys.

- Conceptually:

  ```text id="m9q4vr"
  Client
  
  ↓

  Expected Certificate / Key

  ↓

  Burp-Generated Certificate Does Not Match

  ↓

  Connection Rejected
  ```

- This is particularly relevant in some mobile and specialized application testing environments.

- It is different from simply forgetting to install the Burp CA.

---

## Mutual TLS

- Some applications require the client to present its own certificate.

- Conceptually:

  ```text id="x5n8pt"
  Server Verifies Client Certificate

  +

  Client Verifies Server Certificate
  ```

- This introduces additional configuration beyond ordinary browser HTTPS interception.

- Recognize that TLS can involve more than server certificate trust.

---

## Do Not Disable Security Randomly

- When TLS errors appear, avoid immediately:

  * Disabling certificate verification everywhere.
  * Installing unknown certificates.
  * Changing unrelated proxy options.
  * Enabling unusual compatibility settings.

- Instead diagnose the failing layer.

---

## HTTPS Troubleshooting Ladder

- Use:

  ```text id="q2m7kn"
  1. Does Burp Launch?

  ↓

  2. Is Listener Working?

  ↓

  3. Is Browser Routed Through Burp?

  ↓

  4. Does Request Appear in HTTP History?

  ↓

  5. Does HTTP Work?
  
  ↓

  6. Is Burp CA Trusted by This Browser?

  ↓

  7. Does HTTPS Work?

  ↓

  8. Is the Application Using Special TLS Behavior?
  ```

- This order prevents random configuration changes.

---

## Understanding TLS Errors

- Different TLS errors can indicate different problems.

- Do not treat every error as:

  > "Burp certificate is broken."

- Consider:

  * Wrong proxy type.
  * Incorrect listener configuration.
  * Trust-store issue.
  * Protocol mismatch.
  * Another proxy layer.
  * Application-specific TLS requirements.

- Use evidence from earlier layers.

---

## Safe Certificate Handling

- Treat CA certificates carefully.

- Good practices include:

  ```text id="n4p9mw"
  [ ] Use a dedicated testing browser/profile.

  [ ] Obtain the certificate from your own Burp environment.

  [ ] Import it only where required.
    
  [ ] Do not distribute private CA material.

  [ ] Remove unnecessary trust when retiring the environment.

  [ ] Do not install unknown CA certificates.
  ```

- Trust configuration is security-sensitive.

---

## Verification Test

- After configuring HTTPS trust:

  ```text id="v7m3qx"
  Start Burp

  ↓

  Confirm Listener

  ↓

  Use Testing Browser

  ↓

  Visit an Authorized HTTPS Test Page

  ↓

  Check HTTP History

  ↓

  Confirm HTTPS Request/Response Is Visible
  ```

- If this works without unexpected trust errors, basic HTTPS interception is functioning.

---

## Important Distinction

- Three separate conditions must be satisfied:

  ```text id="k5n8pr"
  Browser Can Reach Burp

  +

  Browser Trusts Burp for HTTPS

  +

  Burp Can Reach Target

  ↓

  Successful HTTPS Proxying  
  ```

- A failure in any one can break the workflow.

---

## Professional Mental Model

- Think of HTTPS troubleshooting as two sides:

  ```text id="r3q9mv"
  Browser

  ↓

  Trusts Burp?
  ```

- and:

  ```text id="p8m2kw"
  Burp

  ↓

  Can Connect to Target?
  ```

- Burp sits between two different connection contexts.

- This mental model becomes increasingly important in advanced testing.

---

## Key Takeaways

* HTTPS encrypts browser-server communication.
* Burp intercepts HTTPS by operating between the client and server connection contexts.
* The testing browser must trust Burp's CA for normal HTTPS interception.
* Installing a CA certificate is a significant trust decision.
* Prefer a dedicated testing browser or profile.
* Certificate trust stores vary across browsers and operating systems.
* HTTP working while HTTPS fails often points toward TLS trust configuration.
* Certificate pinning and mutual TLS are different from ordinary CA trust issues.
* Troubleshoot TLS layer by layer.
* Never install unknown CA certificates or disable security controls blindly.
