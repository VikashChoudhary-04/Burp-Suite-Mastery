# Request and Response Lifecycle

> **Module 04 — How Burp Suite Works**

> **Progress**

```text
█████░░░░░░ 5 / 11 Lessons
```

- Every action performed in a web application eventually becomes one or more HTTP requests and responses.

- When Burp Suite is placed between the client and the target, those messages pass through a sequence of stages before the browser displays the final result.

- A simplified lifecycle is:

  ```text
  User Action

  ↓

  Browser Creates Request

  ↓
    
  Burp Receives Request
  
  ↓

  Request May Be Intercepted / Modified

  ↓

  Burp Sends Request Upstream
  
  ↓
  
  Server Processes Request
  
  ↓  

  Server Creates Response
  
  ↓

  Burp Receives Response
  
  ↓

  Response May Be Inspected / Modified
  
  ↓

  Browser Receives Response
  
  ↓
  
  Browser Processes Result
  ```  

- Understanding this lifecycle is fundamental because Burp's tools operate at different points within it.

- Proxy observes traffic.

- Intercept pauses selected messages.

- HTTP history records traffic.

- Repeater creates independent manual replays.

- Intruder generates repeated modified requests.

- Logger records broader Burp traffic.

- Scanner may create additional testing requests.

- The more accurately you understand the lifecycle, the easier it becomes to understand what Burp is actually doing.

---

## Learning Objectives

- By the end of this lesson, you should be able to:

  * Trace a request from a user action to the server and back.
  * Explain how browsers construct HTTP requests.
  * Understand where Burp enters the lifecycle.
  * Distinguish interception from passive observation.
  * Understand what happens when a request is modified.
  * Explain server-side request processing conceptually.
  * Understand how responses return through Burp.
  * Explain redirects and request chains.
  * Understand how cookies and tokens change across requests.
  * Recognize asynchronous and background requests.
  * Distinguish original traffic from requests generated by Burp tools.
  * Understand why one browser action may create many HTTP messages.
  * Build a reliable mental model for investigating request behavior.

---

## Stage 1 — User Action

- A request often begins with an action such as:

  ```text
  Opening a Page

  Clicking a Button

  Submitting a Form
  
  Changing a Setting

  Uploading a File

  Searching
  
  Logging In

  Loading More Content
  ```  

- Example:

  ```text
  User clicks:

  View Profile
  ```  

- This may cause JavaScript or the browser itself to initiate:

  ```http
  GET /api/profile HTTP/1.1
  Host: example.com
  ```

- The important principle is:

  ```text
  User Action

  ↓
  
  Application Logic

  ↓

  One or More Network Requests
  ```

---

## Not Every Request Requires a Click

- Requests can also be generated automatically.

- Examples:

  ```text
  Page Load

  JavaScript Execution

  Timers

  Background Refresh
  
  Analytics

  WebSocket Setup

  API Polling

  Token Refresh

  Image Loading  
  ```

- Therefore:

  ```text
  No User Click

  ≠

  No HTTP Traffic
  ```

---

## Stage 2 — Browser Constructs the Request

- The browser or application determines:

  ```text
  Method

  Scheme
  
  Hostname

  Port
  
  Path

  Query Parameters

  Headers
  
  Cookies

  Body
  
  Protocol
  ```

- Example:

  ```http  
  POST /api/login HTTP/1.1
  Host: example.com
  Content-Type: application/json
  Cookie: tracking=abc123

  {
    "username": "alice",
    "password": "ExamplePass"
  }
  ```

- The request represents the client's intended operation.

---

## Where Request Data Comes From

- Different request components may come from different sources.

- For example:

  ```text
  Path

  ↓

  Application Route
  ```  

  ```text
  Cookie

  ↓

  Browser Cookie Store
  ```

  ```text
  Authorization Header

  ↓

  Application Authentication Logic
  ```

  ```text
  JSON Body
  
  ↓

  Form Input / JavaScript
  ```

  ```text
  CSRF Token

  ↓

  DOM / Cookie / Previous Response
  ```  

- Understanding where each value originates helps determine whether and how it can be manipulated.

---

## Stage 3 — Browser Sends Traffic Toward Burp

- When the browser is configured to use Burp:

  ```text
  Browser

  ↓

  Burp Proxy Listener
  ```

- instead of following the ordinary direct path:

  ```text
  Browser

  ↓

  Target
  ```  

- For HTTPS, the TLS interception architecture discussed previously applies.

---

## Stage 4 — Burp Receives the Request

- Once Burp receives the request, several things may happen.

- Conceptually:

  ```text
  Incoming Request

  ↓
  
  Burp Proxy Processing

  ├── Scope / Display Handling
  ├── Match and Replace Rules
  ├── Interception Rules
  ├── Extensions
  ├── Logging / History
  └── Other Configured Processing
  ```

- The exact behavior depends on configuration.

---

## Burp Is More Than the Intercept Tab

- A common beginner mental model is:

  ```text
  Request

  ↓
  
  Intercept Tab

  ↓
  
  Server
  ```

- A better model is:

  ```text
  Request

  ↓
  
  Burp Proxy Infrastructure

  ↓
  
  Multiple Processing / Observation Features

  ↓
  
  Server
  ```  

- The Intercept interface is only one part of Burp's request-processing environment.

---

## Stage 5 — Interception Decision

- Burp determines whether the request should pause for manual action.

- Conceptually:

  ```text
  Request

  ↓

  Matches Interception Rules?

  ├── Yes → Pause
  └── No  → Continue Automatically
  ```

- This is why:

  ```text
  Intercept ON
  ```

does not necessarily mean every possible request must always pause.

- Interception behavior can depend on configured rules.

---

## Intercept OFF

- With Intercept OFF:

  ```text
  Browser

  ↓

  Burp

  ↓

  Automatically Forwarded

  ↓

  Server
  ```

- Burp still remains in the traffic path.

- Requests can still appear in HTTP history.

- Therefore:

  ```text
  Intercept OFF

  ≠

  Burp Bypassed
  ```  

---

## Intercept ON

- With a matching request:

  ```text
  Browser

  ↓

  Burp

  ↓

  Request Paused
  ```  

- At this point, the server has generally not yet received that paused request.

- You can:

  ```text
  Inspect

  Modify

  Forward

  Drop
  ```

---

## Why the Browser Appears to Freeze

- Suppose:

  ```text
  Browser sends request

  ↓

  Burp pauses request
  ```

- The browser waits for a response.

- But:

  ```text
  Server has not received request yet
  ```  

- Therefore:

  ```text
  No Server Processing

  ↓  

  No Response

  ↓

  Browser Keeps Waiting
  ```  

- This often looks like the page has frozen.

---

## Stage 6 — Manual Modification

- Suppose the original request contains:

  ```http
  GET /api/orders/105 HTTP/1.1
  Host: example.com
  Cookie: session=abc123
  ```

- While intercepted, you change:

  ```text
  105
  ```

- to:

  ```text
  106
  ```  

- Then forward it.

- The server receives the modified request:

  ```http
  GET /api/orders/106 HTTP/1.1
  Host: example.com
  Cookie: session=abc123
  ```

- not the original version.

---

## Critical Principle

- Once you modify an intercepted request before forwarding:

  ```text
  Browser Intended Request

  ≠
  
  Server Received Request
  ```

- Burp has changed the message in transit.

- This is one of the foundations of manual web security testing.

---

## Change One Variable at a Time

- Suppose you modify:

  ```text
  Order ID

  Cookie

  User-Agent

  Authorization Header
  ```  

simultaneously.

- Then the response changes.

- You cannot easily determine which modification caused the difference.

- Prefer:

  ```text
  Baseline Request

  ↓

  Change One Variable

  ↓

  Observe Response

  ↓

  Record Result

  ↓

  Change Next Variable
  ```

- This creates interpretable evidence.

---

## Stage 7 — Forward or Drop

- If you select:

  ```text
  Forward
  ```

the request continues toward the target.

- Conceptually:

  ```text
  Burp

  ↓

  Route / Resolve Destination

  ↓

  Establish or Reuse Connection

  ↓

  Send Request
  ```  

- If you select:

  ```text
  Drop
  ```  

- Burp does not forward that intercepted message normally to the destination.

---

## What Happens When You Drop a Request?

- Conceptually:

  ```text
  Browser

  ↓

  Burp

  ↓

  DROP

  X

  Server
  ```

- The target does not receive that request through the normal flow.

- The browser may:

  * Wait.
  * Show an error.
  * Retry.
  * Continue with other requests.
  * Trigger alternative application logic.

- Behavior depends on the application.

---

## Dropping One Request Does Not Stop Everything

- Modern pages generate many requests.

- Suppose:

  ```text
  GET /dashboard

  GET /app.js

  GET /api/profile

  GET /api/messages
  ```  

- You drop:

  ```text
  GET /api/messages
  ```  

- The others may still proceed.

- The page may partially load.

- This makes dropping requests useful for understanding dependencies.

---

## Stage 8 — Burp Sends the Request Upstream

- After forwarding:

  ```text
  Burp

  ↓

  Target Infrastructure
  ```

- Burp may:

  * Reuse an existing connection.
  * Open a new connection.
  * Use HTTP/1.1.
  * Use HTTP/2.
  * Route through an upstream proxy.
  * Route through SOCKS.
  * Use TLS for HTTPS.
  * Apply configured routing rules.

- The browser does not directly control every detail of this upstream connection.

---

## Stage 9 — Target Infrastructure Receives the Request

- The request may not go directly to the application code.

- A modern architecture may look like:

  ```text
  Burp

  ↓

  CDN

  ↓

  WAF

  ↓
  
  Load Balancer

  ↓

  Reverse Proxy

  ↓

  Application Server

  ↓

  Database / Internal Services
  ```  

- The exact architecture varies.

---

## Why Infrastructure Matters

- A response may be generated by:

  ```text
  CDN

  WAF
  
  Reverse Proxy
  
  Application
  
  Authentication Gateway

  API Gateway
  ```

- Therefore:

  ```text
  Response From Target Domain

  ≠

  Necessarily Response From Application Code
  ```

- This distinction matters during investigation.

---

## Example — WAF Response

- You send:

  ```http
  GET /search?q=<test> HTTP/1.1
  Host: example.com
  ```  

- and receive:

  ```text
  403 Forbidden
  ```

- Possible source:

  ```text
  WAF
  ```  

- rather than:

  ```text
  Application
  ```  

- Do not automatically conclude that the application itself processed and rejected the input.

---

## Stage 10 — Application Processing

- If the request reaches the application, it may perform operations such as:

  ```text
  Parse Input

  Authenticate User
  
  Check Authorization

  Validate CSRF

  Apply Business Logic

  Query Database

  Call Internal API

  Generate Response
  ```  

- Conceptually:

  ```text
  HTTP Request

  ↓

  Routing

  ↓

  Authentication

  ↓

  Authorization

  ↓

  Business Logic

  ↓

  Data Access

  ↓

  Response Generation
  ```  

- The order varies by application.

---

## Security Testing Focus

- Many vulnerabilities occur because one of these stages behaves incorrectly.

- Examples:

  ```text
  Authentication

  ↓
  
  Can identity checks be bypassed?
  ```

  ```text
  Authorization

  ↓

  Can User A access User B's object?
  ```

  ```text
  Business Logic

  ↓

  Can workflow assumptions be violated?
  ```

  ```text
  Input Handling

  ↓

  Can malicious input reach an unsafe sink?
  ```

- Burp lets you manipulate the request that enters this processing chain.

---

## Stage 11 — Server Generates a Response

- Example:

  ```http
  HTTP/1.1 200 OK
  Content-Type: application/json
  Set-Cookie: session=xyz789

  {
    "id": 105,
    "owner": "alice"  
  }
  ```

- The response may contain:

  ```text
  Status Code

  Headers

  Cookies

  Body

  Redirect Instructions

  Cache Information

  Security Headers
  ```  

- Each can provide useful evidence.

---

## Stage 12 — Response Returns to Burp

- The reverse path begins:

  ```text
  Server

  ↓

  Burp

  ↓

  Browser
  ```  

- For HTTPS:

  ```text
  Server

  ↓
  
  TLS to Burp

  ↓

  Burp Decrypts Response

  ↓

  Response Available for Inspection

  ↓

  Burp Encrypts Toward Browser

  ↓

  Browser
  ```  

- Burp can therefore inspect responses as well as requests.

---

## Response Interception

- Burp can be configured to intercept responses under appropriate conditions.

- Conceptually:

  ```text
  Server Response

  ↓

  Burp

  ↓

  Pause

  ↓

  Inspect / Modify

  ↓

  Forward

  ↓

  Browser
  ```  

- This can help demonstrate how the browser reacts to changed server responses.

---

## Important Limitation of Response Modification

- Suppose the real server sends:

  ```json
  {  
    "role": "user"
  }
  ```

- You modify the response in Burp to:

  ```json
  {
    "role": "admin"
  }
  ```

- The browser may display admin-related UI.

- But this alone does **not** prove server-side privilege escalation.

- You changed what the client saw.

- You did not necessarily change server-side authorization state.

---

## Client-Side Appearance vs Server-Side Reality

- Always distinguish:

  ```text
  Modified Response

  ↓
  
  Browser Displays Different UI
  ```

- from:

  ```text
  Server Actually Grants New Privileges
  ```

- A real authorization issue requires evidence that protected server-side actions or data became accessible improperly.

---

## Stage 13 — Browser Processes the Response

- The browser may:

  ```text
  Render HTML

  Execute JavaScript

  Store Cookies
  
  Follow Redirects

  Load Images

  Call APIs

  Update DOM

  Cache Resources

  Trigger More Requests
  ```

- Therefore the response may immediately create another request lifecycle.

---

## Response Creates New Requests

- Example response:

  ```html
  <script src="/app.js"></script>
  <img src="/logo.png">
  ```

- Browser then generates:

  ```text
  GET /app.js

  GET /logo.png
  ```  

- JavaScript may then generate:

  ```text
  GET /api/profile

  GET /api/messages
  ```

- One initial request can create a large request tree.

---

## Request Chains

- A web interaction is often better understood as a chain rather than a single request.

- Example:

  ```text
  GET /login

  ↓

  200 Login Page

  ↓

  POST /login

  ↓

  302 /dashboard

  ↓

  GET /dashboard

  ↓

  200 HTML

  ↓

  GET /api/profile

  ↓

  GET /api/notifications
  ```  

- Analyzing only one message may miss the actual workflow.

---

## Redirect Lifecycle

- Suppose the browser sends:

  ```http
  POST /login HTTP/1.1
  Host: example.com
  ```

- Server responds:

  ```http
  HTTP/1.1 302 Found
  Location: /dashboard
  Set-Cookie: session=abc123
  ```

- The browser may then send:

  ```http
  GET /dashboard HTTP/1.1
  Host: example.com
  Cookie: session=abc123
  ```

- This is a new request.

---

## Important Redirect Mental Model

- Do not think:

  ```text
  302 Response

  ↓

  Server Automatically Sends Dashboard Page
  ```  
  
- Instead:

  ```text
  Server Sends 302

  ↓

  Browser Reads Location

  ↓

  Browser Creates New Request
  
  ↓

  Server Processes New Request
  ```
  
- This distinction is important when testing authentication flows.

---

## Cookies Across the Lifecycle

- Suppose the response contains:

  ```http
  Set-Cookie: session=abc123; Secure; HttpOnly
  ```

- The browser may store the cookie.

- Later:

  ```http
  GET /dashboard HTTP/1.1
  Host: example.com
  Cookie: session=abc123
  ```

- The lifecycle becomes:

  ```text
  Server Response

  ↓

  Set-Cookie
  
  ↓

  Browser Cookie Store

  ↓
  
  Future Request
  
  ↓

  Cookie Header
  ```
  
---

## Burp Does Not Replace Browser State Automatically in Every Tool

- Normal browser traffic uses the browser's state management.

- But when you send a request to another Burp tool such as Repeater:

  ```text
  Browser Request

  ↓
  
  Send to Repeater
  
  ↓

  Independent Manual Request
  ```

- the request becomes a separate testing artifact.

- Changes in the browser do not necessarily update old Repeater tabs automatically.

---

## Stale Session Example

- You send this to Repeater:

  ```http
  Cookie: session=old123
  ```

- Then log in again through the browser.

- Browser now uses:

  ```http
  Cookie: session=new456
  ```

- The existing Repeater request may still contain:

  ```http
  Cookie: session=old123
  ```

- Result:

  ```text
  Browser Works

  but

  Repeater Gets 401
  ```  

- Always inspect current authentication state.

---

## CSRF Tokens Across Requests

- A workflow may use:

  ```text
  GET /settings

  ↓

  Response Contains CSRF Token
  
  ↓

  Browser Submits Token
  
  ↓

  POST /settings
  ```

- If you replay an old request later, the token may be:

  ```text
  Expired

  Single-Use
  
  Session-Bound

  Rotated
  ```
  
- A failed replay does not automatically mean the endpoint is secure against every test.

- Understand the token lifecycle.

---

## Authentication Token Refresh

- Modern applications may automatically refresh access tokens.

- Example:

  ```text
  Access Token Expires

  ↓
  
  POST /token/refresh
  
  ↓
  
  New Access Token Returned

  ↓

  Application Retries API Request
  ```  

- In HTTP history you may see:

  ```text
  401

  ↓
  
  Refresh Request
  
  ↓

  200
  
  ↓

  Original API Request Retried

  ↓
  
  200
  ```  

- Without understanding the lifecycle, this can appear confusing.

---

## Background API Requests

- Single-page applications often communicate heavily through APIs.

- The visible page may remain unchanged while requests occur:

  ```text
  GET /api/me

  GET /api/feed
  
  POST /api/events

  GET /api/notifications
  ```  

- Therefore:

  ```text
  Browser UI

  ≠
  
  Complete Application Activity
  ```

- HTTP history often reveals much more.

---

## Asynchronous Requests

- JavaScript can send requests without full-page navigation.

- Examples:

  ```text
  fetch()

  XMLHttpRequest
  
  GraphQL Requests

  Background API Calls
  ```  

- Conceptually:

  ```text
  JavaScript

  ↓

  HTTP Request

  ↓

  Server

  ↓

  JSON Response

  ↓
  
  JavaScript Updates Page
  ```

- The browser may never visibly reload.

---

## Race Conditions and Concurrent Requests

- Not every lifecycle is strictly sequential.

- Modern applications may send requests concurrently.

- Conceptually:

  ```text
  Browser

  ├── Request A ─────►
  ├── Request B ─────►
  └── Request C ─────►
  ```

- Responses may return in a different order:

  ```text
  Response B

  Response A
  
  Response C
  ```  

- This matters when investigating:

  * Race conditions.
  * State transitions.
  * Duplicate actions.
  * Parallel API behavior.

---

## One Request Can Trigger Server-Side Actions

- A single request may cause the server to perform:

  ```text
  Database Write

  Email Generation

  Queue Job
  
  Webhook

  Internal API Call

  Payment Operation
  ```

- The immediate HTTP response may not reveal every resulting action.

- Therefore:

  ```text
  HTTP Response

  ≠
  
  Complete Server-Side Impact
  ```  

- This becomes important in out-of-band testing and business logic analysis.

---

## Burp HTTP History

- HTTP history records requests and responses passing through the proxy according to Burp's behavior and configuration.

- It helps reconstruct:

  ```text
  What Was Requested?

  When?

  To Which Host?

  With Which Method?

  What Was the Response?

  What Status Code?

  What Parameters Were Present?
  ```

- This makes it a core investigation tool.

---

## History Is Evidence, Not Interpretation

- Suppose you see:

  ```text
  GET /admin

  ↓
  
  200 OK
  ```  

- Do not immediately conclude:

  ```text
  Unauthorized Admin Access
  ```

- The response could be:

  * Login page.
  * Generic SPA shell.
  * Error message with status 200.
  * Redirect content.
  * Public page.

- Inspect the actual response and application behavior.

---

## Repeater Creates a New Request Lifecycle

- Suppose:

  ```text
  Browser

  ↓

  Proxy History

  ↓

  Send to Repeater
  ```

- Now Repeater can send:

  ```text
  Repeater

  ↓

  Burp

  ↓

  Target
  ```  

- This is not the browser replaying the request.

- Repeater is now the client for that manual request.

---

## Repeater Mental Model

```text
Original Browser Request

↓

Copied into Repeater

↓

Independent Editable Request

↓

New Request Sent to Server

↓

New Response
```

- Changing Repeater does not automatically change what the browser originally sent.

---

## Intruder Creates Many New Lifecycles

- Conceptually:

  ```text
  Base Request

  ↓
  
  Payload 1 → Request 1
  
  Payload 2 → Request 2

  Payload 3 → Request 3

  Payload 4 → Request 4
  ```  

- Each request is independently processed by the target.

- This is why automated testing must consider:

  * Rate limits.
  * Account lockouts.
  * State changes.
  * Server load.
  * Scope authorization.

---

## Scanner Can Generate Additional Traffic

- Burp Scanner may create requests that the browser never generated.

- Conceptually:

  ```text
  Observed Request

  ↓
  
  Scanner Analysis

  ↓

  Additional Audit Requests

  ↓

  Target
  ```  

- Therefore:

  ```text
  Traffic in Burp

  ≠
  
  Always Human Browser Traffic
  ```

- Know which tool generated a request before interpreting it.

---

## Extensions Can Affect the Lifecycle

- Burp extensions may:

  ```text
  Observe Requests

  Modify Requests

  Generate Requests

  Analyze Responses

  Add Headers

  Perform Active Tests
  ```  

- Therefore an advanced Burp setup may behave differently from a clean installation.

- When troubleshooting unexpected traffic:

  ```text
  Check Extensions
  ```

as well as proxy settings.

---

## Match and Replace

- Configured rules can automatically transform traffic.

- Conceptually:

  ```text
  Incoming Request

  ↓

  Rule Matches Header

  ↓

  Header Modified

  ↓

  Request Forwarded
  ```  

- This can be useful, but forgotten rules can create confusing behavior.

- Example:

  ```text
  Browser Sends:

  User-Agent: Firefox
  ```
  
- Burp rule changes it to:

  ```text
  User-Agent: TestingClient
  ```

- The server never sees the original value.

---

## Baseline Requests

- Before modifying a request, establish a baseline.

- Record:

  ```text
  Request

  Status Code

  Response Length

  Important Headers

  Response Body

  Timing
  
  Application State
  ```

- Then modify one variable.

- Compare:

  ```text
  Baseline

  vs
  
  Modified Request
  ```

- This is the foundation of controlled manual testing.

---

## Response Differences

- Useful differences may include:

  ```text
  Status Code

  Content Length

  JSON Fields

  Redirect Location

  Error Message

  Headers

  Response Time

  Application State
  ```  

- No single difference automatically proves a vulnerability.

- Interpret changes in context.

---

## Example — IDOR Investigation Lifecycle

- Baseline:

  ```http
  GET /api/orders/105 HTTP/1.1
  Cookie: session=userA
  ```

- Response:

  ```text
  200

  Order belongs to User A
  ```

- Test:

  ```http
  GET /api/orders/106 HTTP/1.1
  Cookie: session=userA
  ```

- Possible response:

  ```text
  200

  Order belongs to User B
  ```  

- Now investigate:

  ```text
  Did server return unauthorized data?

  Is object actually owned by another user?

  Was authentication state unchanged?

  Can action be reproduced?

  What is the impact?
  ```  

- The lifecycle provides evidence.

---

## Example — Authentication Test

- Original:

  ```http
  GET /account HTTP/1.1
  Cookie: session=abc123
  ```

- Baseline:

  ```text
  200 Account Page
  ```

- Modified:

  ```http
  GET /account HTTP/1.1
  ```

- Possible response:

  ```text
  302 /login
  ```

- Interpretation:

  ```text
  Removing session changed authentication behavior.
  ```

- This is evidence of expected access control behavior, not a vulnerability.

---

## Example — Client-Side Parameter

- Request:

  ```http
  POST /purchase HTTP/1.1
  Content-Type: application/json

  {
    "product_id": 50,
    "price": 100
  }
  ```

- You change:

  ```text
  price: 100
  ```

- to:

  ```text
  price: 1
  ```  

- Possible outcomes:

  ```text
  Server Rejects

  Server Recalculates Price
  
  Server Accepts Modified Price
  ```

- The response alone may not be sufficient.

- Verify the resulting server-side transaction state.

---

## State-Changing Requests

- Be careful with requests such as:

  ```text
  POST /purchase

  DELETE /account

  POST /transfer

  POST /invite

  POST /email/change
  ```  

- Replaying them may cause real actions.

- Before sending:

  ```text
  Understand Side Effects
  ```

especially during professional assessments.

---

## Safe Workflow

- For state-changing requests:

  ```text
  Understand Endpoint

  ↓

  Confirm Authorization / Scope

  ↓

  Use Controlled Test Data

  ↓

  Establish Baseline

  ↓

  Send Minimal Test

  ↓

  Verify Resulting State

  ↓

  Document Evidence
  ```  

- Do not blindly replay requests.

  ```text

  Server Processing

  ↓

  Response Received
  ```  

## Cached Responses

- Not every response necessarily reflects fresh application processing.

- Possible cache layers:

  ```text
  Browser Cache

  CDN Cache
  
  Reverse Proxy Cache

  Application Cache
  ```

- A response may be served without reaching the expected backend logic.

- When testing behavior, consider whether caching affects results.

## 304 Not Modified

- A browser may send conditional headers such as:

  ```http
  If-None-Match: "abc123"
  ```

- Server may return:

  ```http
  HTTP/1.1 304 Not Modified
  ```

- The browser can reuse cached content.

- This can make response analysis confusing if you expect a full body.

---

## Service Workers

- Modern web applications may use service workers.

- Conceptually:

  ```text
  Browser Page

  ↓

  Service Worker

  ↓

  Network / Cache Logic  
  ```  

- A service worker can influence how requests are handled.

- This may create differences between expected page behavior and network traffic.

- For security testing, focus on what actually reaches the server and what is handled locally.

---

## WebSockets

- Not all modern application communication follows repeated independent HTTP request-response cycles.

- A WebSocket commonly begins with an HTTP-based handshake and then becomes a persistent bidirectional communication channel.

- Conceptually:

  ```text
  HTTP Handshake

  ↓

  Protocol Upgrade

  ↓

  Persistent WebSocket Connection

  ↕
  Messages
  ```

- WebSocket lifecycle is covered separately in advanced Burp modules.

---

## HTTP/2 Lifecycle Differences

- HTTP/2 can multiplex multiple streams over one connection.

- Conceptually:

  ```text
  One Connection

  ├── Request Stream 1
  ├── Request Stream 3
  └── Request Stream 5
  ```

- Responses can progress independently.

- The logical model remains:

  ```text
  Request

  ↓

  Processing

  ↓

  Response
  ```  

but network ordering and connection behavior can be more complex.

## Browser Retry Behavior

- A browser or application may retry requests automatically.

- Possible reasons:

  ```text
  Network Failure

  Authentication Refresh

  Application Retry Logic

  Service Worker

  JavaScript Error Handling
  ```  

- Therefore duplicate requests in history do not automatically mean the user clicked twice.

---

## Distinguish Cause From Observation

- Suppose you observe:

  ```text
  Two identical POST requests
  ```

- Possible explanations:

  ```text
  User submitted twice

  Browser retried

  JavaScript sent twice

  Extension generated request

  Scanner generated request

  Application workflow intentionally duplicated action
  ```  

- Investigate before concluding.

## Request Lifecycle Investigation Method

- For any interesting behavior, answer:

  ```text
  1. What triggered the request?

  2. Which client/tool generated it?
  
  3. What was the exact baseline request?

  4. Was it intercepted?

  5. Was it modified?

  6. What exact request reached the target?
  
  7. Which infrastructure likely handled it?
  
  8. What response came back?
  
  9. Did Burp modify the response?

  10. What did the browser/application do next?

  11. Did the response trigger more requests?

  12. Did server-side state actually change?
  ```  

- This turns random experimentation into structured investigation.

## Common Mistakes

- Avoid:

```text
Intercept OFF means Burp is not involved.
```

- Incorrect.

---

```text
The request shown in the browser is always exactly what the server received.
```

- Incorrect if a proxy, extension, rule, or tool modified it.

---

```text
One click equals one request.
```

- Incorrect for modern applications.

---

```text
A 200 response always means success.
```

- Incorrect.

- Inspect the content and resulting state.

---

```text
A 403 always came from application authorization logic.
```

- Incorrect.

- Infrastructure such as a WAF may have generated it.

---

```text
Changing the response changes server-side authorization.
```

- Incorrect.

- It may only change client-side behavior.

---

```text
Repeater is simply controlling the browser.
```

- Incorrect.

- Repeater sends independent requests.

---

```text
Every request in Burp came from the browser.
```

- Incorrect.

- Scanner, Intruder, extensions, and other tools may generate traffic.

---

```text
If a replay fails, the endpoint cannot be tested.
```

- Incorrect.

- Session state, CSRF tokens, signatures, timestamps, or workflow dependencies may have changed.

---

```text
A changed response automatically proves a vulnerability.
```

- Incorrect.

- It proves only that behavior changed until impact is validated.

## Practical Exercise

- Use an authorized practice application.

### Exercise 1 — Trace One Page Load

1. Clear or filter HTTP history.
2. Load one page.
3. Identify the initial document request.
4. Identify additional JavaScript, CSS, image, and API requests.
5. Determine which hosts received traffic.

  ```text

  ↓
  
  Many Requests
  ```

1. Enable Intercept.
2. Trigger a harmless request.

4. Modify one harmless header.
5. Forward the request.
6. Observe the response.

  ```text

  Modified Value

  Response Difference
  ```

- Find a login or navigation flow that produces a redirect.

  ```text

  ↓

  New Browser Request

  ↓
  ```  

## Key Takeaways

* Every web interaction consists of one or more message lifecycles.
* A browser action may generate many requests, including asynchronous background traffic.
* When configured as a proxy, Burp receives traffic before forwarding it upstream.
* Intercept OFF does not bypass Burp; traffic can still pass through the proxy and appear in history.
* Intercept ON can pause matching requests before the server receives them.
* If an intercepted request is modified, the server receives the forwarded modified version.
* Dropping one request does not necessarily stop other requests generated by the page.
* Modern target infrastructure may include CDNs, WAFs, gateways, reverse proxies, and application servers.
* Status codes alone do not prove application behavior or vulnerabilities.
* Responses can set cookies, redirect browsers, execute JavaScript, and trigger additional requests.
* Redirects create new HTTP requests rather than magically transferring the original request.
* Authentication and CSRF state may change over time, making old replays stale.
* Repeater, Intruder, Scanner, and extensions can generate requests independently of the browser.
* Response modification may alter client-side behavior without changing server-side authorization.
* Establishing a baseline and changing one variable at a time produces stronger evidence.
* State-changing requests should be replayed carefully and only within authorized testing.
* Professional testing requires tracing what triggered a request, what reached the server, what returned, and what state actually changed.
